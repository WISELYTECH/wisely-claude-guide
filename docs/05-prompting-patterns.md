# 프롬프팅 패턴

> 같은 질문도 어떻게 하느냐에 따라 결과가 달라집니다. Claude에게 효과적으로 요청하는 방법을 소개합니다.

---

## 핵심 원칙: 구체적일수록 좋다

| 모호한 요청 | 구체적인 요청 |
|-------------|-------------|
| 코드 정리해줘 | `UserService.ts`의 `createUser` 함수에서 중복된 유효성 검사를 하나로 합쳐줘 |
| 리포트 만들어줘 | 이번 달 매출 CSV를 분석해서 주간별 트렌드 차트와 상위 5개 상품을 포함한 엑셀로 만들어줘 |
| 이거 고쳐줘 | 로그인 시 비밀번호 틀렸을 때 에러 메시지가 안 뜨는 버그를 수정해줘 |
| 테스트 작성해줘 | `calculateDiscount` 함수의 엣지 케이스(0원, 음수, 100% 할인)에 대한 단위 테스트를 작성해줘 |

**왜?** Claude는 모호한 요청을 받으면 추측해서 진행합니다. 추측이 맞으면 좋지만, 틀리면 처음부터 다시 해야 합니다. 구체적으로 요청하는 30초가 재작업 30분을 줄여줍니다.

---

## 패턴 1: 제약 조건 먼저 말하기

결과물의 형식, 언어, 분량, 톤을 먼저 명시하면 원하는 결과에 가까워집니다.

```
한국어로, 테이블 형식으로, 각 항목 2줄 이내로 정리해줘.
```

```
Slack에 공유할 거라 3문장 이내로 요약해줘.
```

```
비개발자도 이해할 수 있게 기술 용어 없이 설명해줘.
```

**제약 조건 체크리스트:**
- **형식**: 테이블, 불릿, 코드블록, 엑셀, 마크다운
- **분량**: 몇 줄, 몇 페이지, 몇 문장
- **언어/톤**: 한국어, 캐주얼, 보고서체
- **대상 독자**: 경영진, 개발자, 신입, 고객

---

## 패턴 2: 컨텍스트 제공하기

Claude는 여러분의 상황을 모릅니다. 배경을 알려줄수록 정확한 답이 나옵니다.

**나쁜 예:**
```
멤버십 기능 구현해줘.
```

**좋은 예:**
```
우리 서비스는 3단계 멤버십(Silver, Gold, Platinum)이 있어.
현재 등급은 수동으로 변경하고 있는데,
월 구매금액 기준으로 자동 업그레이드하는 기능을 만들고 싶어.
기준은 Silver: 0원, Gold: 5만원, Platinum: 20만원이야.
```

**팁**: "이 파일을 읽어봐"로 시작하면 Claude가 맥락을 먼저 파악합니다.
```
commerce-web-api/src/membership/membership.service.ts를 읽고,
현재 등급 변경 로직이 어떻게 되어 있는지 설명해줘.
```

---

## 패턴 3: 단계 나누기 (Research → Plan → Execute)

큰 작업을 한 번에 시키면 Claude가 잘못된 방향으로 오래 달릴 수 있습니다. 단계를 나누면 중간에 방향을 잡아줄 수 있습니다.

### Research (조사)
```
현재 결제 모듈이 어떤 PG사를 지원하는지 코드에서 찾아서 정리해줘.
아직 수정하지 말고 현황만 알려줘.
```

### Plan (계획)
```
조사 결과를 바탕으로 토스페이를 추가하려면 어떤 파일을 수정해야 하는지 계획을 세워줘.
아직 코드는 건드리지 마.
```

### Execute (실행)
```
이 계획대로 진행해줘.
```

**"아직 실행하지 마"가 핵심입니다.** 이 한 마디가 Claude가 조사 단계에서 멈추게 합니다.

---

## 패턴 4: 인터뷰 요청하기

요구사항이 머릿속에 있지만 정리가 안 될 때, Claude에게 질문하게 하세요.

```
신규 고객 온보딩 플로우를 기획하고 있어.
나에게 인터뷰해줘 — 대상 사용자, 단계별 화면, 이탈 방지 전략,
엣지 케이스에 대해 충분히 파악될 때까지 질문하고,
끝나면 완전한 기획 스펙을 정리해줘.
```

Claude가 한 번에 하나씩 질문하며 요구사항을 구체화해줍니다.

---

## 패턴 5: 예시 보여주기

원하는 결과물의 형태를 직접 보여주면 가장 정확합니다.

```
커밋 메시지를 이런 형식으로 작성해줘:

feat: 멤버십 자동 업그레이드 기능 추가

- 월 구매금액 기준 등급 자동 변경
- Silver/Gold/Platinum 3단계
- 매월 1일 배치 실행
```

```
API 응답을 이 형식에 맞춰줘:
{
  "success": true,
  "data": { ... },
  "error": null
}
```

## 패턴 6: 피드백 주기

결과가 마음에 안 들 때, "다시 해줘"보다 **뭐가 어떻게 다르면 좋겠는지** 말해주세요.

| 비효과적 | 효과적 |
|----------|--------|
| 다시 해줘 | 톤이 너무 딱딱해. 동료에게 말하듯 캐주얼하게 바꿔줘 |
| 이거 아닌데 | 엑셀 피벗 테이블 대신 단순 합계 테이블로 바꿔줘 |
| 더 잘 해줘 | 각 항목에 구체적인 수치와 근거를 추가해줘 |
| 너무 길어 | 핵심 3가지만 불릿으로 정리해줘 |

**팁**: Claude가 잘한 부분은 유지하라고 말해주세요.
```
전체 구조는 좋은데, 2번 항목의 예시를 우리 서비스에 맞는 걸로 바꿔줘.
```

---

## 패턴 7: 역할 부여하기

특정 관점에서 검토가 필요할 때 역할을 지정하세요.

```
시니어 백엔드 개발자 관점에서 이 API 설계를 리뷰해줘.
성능, 보안, 확장성 측면에서 문제가 있는지 확인해줘.
```

```
QA 엔지니어 관점에서 이 기능의 테스트 케이스를 작성해줘.
정상 케이스, 경계값, 에러 케이스를 모두 포함해줘.
```

```
신입 개발자에게 설명한다고 생각하고,
이 코드의 동작 흐름을 단계별로 설명해줘.
```

## 안티패턴: 이렇게 하면 결과가 나빠집니다

### 한 번에 너무 많이 시키기
```
# 나쁨
결제 모듈 리팩터링하고 테스트 작성하고 문서화하고 PR도 만들어줘.

# 좋음
결제 모듈의 중복 코드를 정리해줘. 끝나면 다음 단계를 알려줄게.
```

### 되돌릴 수 없는 작업을 한 번에 승인하기
```
# 위험
DB 마이그레이션 만들고 바로 실행해줘.

# 안전
DB 마이그레이션 파일만 먼저 만들어줘. 내가 확인 후 실행할게.
```

### 맥락 없이 "이전이랑 같이 해줘"
```
# 나쁨 (새 세션에서)
아까처럼 해줘.

# 좋음
지난번에 commerce-web-api에서 했던 것처럼
NestJS 서비스 레이어에 유효성 검사를 추가해줘.
```

새 세션에서는 이전 대화를 기억하지 못합니다. 필요한 맥락을 다시 알려주세요.
